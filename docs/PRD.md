# 基于 SpringBoot 的心理健康咨询系统（MVP）PRD

## 1. 文档信息
- 产品名称：心理健康咨询系统（前后端分离）
- 目标版本：MVP v1.0
- 技术栈：Vue3 + Element Plus；SpringBoot + Spring Security + MyBatis + MySQL；JWT
- 角色范围：学生 / 咨询师（心理老师）/ 管理员

## 2. 背景与问题
高校心理健康服务存在覆盖面不足、流程分散、隐私顾虑强等问题。通过构建数字化咨询系统，形成“测评筛查—预约咨询—咨询跟进—后台管控”的闭环，提升服务触达与流程效率，并提供可解释的安全与权限控制能力。

## 3. 目标与成功指标
### 3.1 目标
- 搭建可运行稳定的前后端分离咨询系统，支撑三角色协同使用
- 打通三条核心闭环：匿名咨询、心理测评、预约咨询
- 具备最小安全能力：鉴权、授权、数据范围控制、审计留痕

### 3.2 成功指标（验收口径）
- 闭环可演示：咨询（发起/回复/结束）、测评（答题/评分/结果/历史）、预约（排班/预约/取消/完成）全流程无阻断
- 权限正确：未登录访问受保护接口返回 401；越权访问返回 403；数据范围符合角色约束
- 数据一致：同一咨询师同一时段不可重复预约；预约状态流转正确
- 稳定性：核心页面/核心接口无明显报错；本地演示环境下关键操作可在 3 步内完成

## 4. 用户画像与使用场景
### 4.1 学生
- 场景：情绪困扰需要倾诉/建议；希望匿名发起咨询；进行自测并查看建议；预约咨询师线下/线上咨询

### 4.2 咨询师（心理老师）
- 场景：查看学生咨询请求并回复；管理自己的排班与预约；查看与自己关联的学生测评结果与历史
- 约束：在匿名咨询场景下，咨询师可查看学生真实身份（内部可见），学生端仍保持匿名体验

### 4.3 管理员
- 场景：管理账号与角色；查看与监管咨询/预约/测评记录；查看基础统计概览；处理违规内容（下架/删除）

## 5. 范围与边界
### 5.1 必做范围（本 PRD 覆盖）
1) 角色与权限（RBAC）  
2) 用户与安全（登录、JWT、Spring Security、基础审计）  
3) 匿名咨询（留言式对话）  
4) 心理测评（固定量表、固定计分规则，不可配置）  
5) 预约管理（无审核）+ 站内通知  
6) 基础后台管理（用户、记录、统计；不做学院维度）  

### 5.2 明确不做（v1.0 不包含）
- 辅导员/班主任等扩展角色
- 实时聊天（WebSocket）与音视频咨询
- 量表/计分规则后台可配置（量表固定）
- 学院/班级维度统计与组织架构管理
- 短信/邮件提醒（仅站内通知）
- PDF 导出报告

## 7. 功能需求（按模块）
### 7.1 用户与安全
**功能点**
- 账号注册（学生注册、咨询师注册）
- 账号登录、退出
- JWT 鉴权；基于角色的接口授权
- 密码加密存储（BCrypt）
- 管理员可启用/禁用账号、重置密码
- 基础审计日志：记录关键操作（登录、创建/修改/删除核心业务数据、内容下架）
- 安全治理：管理员下架违规咨询会话/消息，并在学生/咨询师端隐藏或提示“内容已下架”

**注册逻辑**
- 学生注册：提供用户名、密码、昵称等基础信息，即刻生效。
- 咨询师注册：提供真实姓名、职称、擅长领域、个人简介等信息，即刻生效（MVP 简化处理）。

**验收标准**
- 未登录访问受保护接口返回 401
- 登录但无权限访问返回 403
- 密码不以明文存储与传输（除登录请求外）
- 禁用账号无法登录

### 7.2 匿名咨询（留言式）
**数据结构（概念）**
- 咨询线程：thread（学生、咨询师、状态、创建时间）
- 咨询消息：message（线程ID、发送者角色、内容、时间）

**咨询归属规则**
- 学生发起咨询时必须选择咨询师，系统以“学生选择的咨询师”作为线程归属咨询师
- 咨询师仅可查看“归属为自己”的线程；管理员可查看全量线程

**学生端**
- 发起咨询（可匿名展示）：填写主题/内容，提交后生成线程
- 查看咨询列表与详情：显示状态、最后回复时间
- 追问：在同一线程下追加消息

**咨询师端**
- 查看分配/关联的咨询线程列表
- 回复消息
- 标记线程状态：未处理 / 处理中 / 已结束
- 可查看学生真实身份（内部可见）

**管理员端**
- 查看全量咨询线程与消息
- 下架/删除违规内容（最小治理能力）

**验收标准**
- 学生仅能看到自己的咨询线程
- 咨询师仅能看到与自己关联的线程
- 线程状态流转正确，已结束线程不允许学生继续追问

### 7.3 心理测评（固定量表）
**量表范围（v1.0）**
- PHQ-9（抑郁筛查）
- GAD-7（焦虑筛查）

**功能点**
- 学生选择量表进入答题页，提交后自动评分并输出风险等级与建议文案
- 学生可查看历史测评记录与单次详情
- 咨询师可查看与自己关联学生的测评记录（用于辅助咨询）
- 管理员可查看测评记录与基础统计概览

**计分与等级**
- 固定规则：按量表标准计分；按阈值区分等级（轻度/中度/重度等）
- v1.0 不提供后台阈值配置能力

**验收标准**
- 提交测评后返回：量表名称、总分、等级、建议、提交时间
- 学生端仅可查看本人测评历史
- 咨询师端仅可查看关联学生测评历史

### 7.4 预约管理（无审核）+ 站内通知
**排班（咨询师端）**
- 配置可预约时间段（日期 + 时间段 + 可预约/不可预约）

**预约（学生端）**
- 选择咨询师 + 选择时间段提交预约
- 咨询师列表支持搜索与分页，并可提示“可预约/已下线（无可选时间）”
- 自动确认并占用时段
- 取消预约
- 查看预约列表与详情

**预约（咨询师端）**
- 查看自己的预约列表
- 标记已完成（可附备注，v1.0 可选）

**站内通知**
- 预约成功通知（学生、咨询师）
- 预约取消通知（学生、咨询师）
- 咨询师回复咨询的提醒（学生）

**验收标准**
- 同一咨询师同一时段不能重复预约
- 取消后释放时段
- 站内通知在对应事件触发后可在消息列表中看到

### 7.6 状态与枚举（v1.0）
**咨询线程状态**
- 未处理：学生已提交咨询，咨询师未回复
- 处理中：咨询师已回复或线程有往来消息
- 已结束：咨询师结束线程，学生不可继续追问

**排班时段状态**
- 可预约：可被学生预约
- 已占用：已生成有效预约
- 不可用：咨询师手动关闭或过期不可预约

**预约状态**
- 已确认：预约创建成功并占用时段
- 已取消：学生取消或管理员取消
- 已完成：咨询师标记完成

**站内通知类型**
- APPOINTMENT_CREATED：预约成功
- APPOINTMENT_CANCELED：预约取消
- CONSULTANT_REPLIED：咨询师回复咨询
- STUDENT_MESSAGE：学生发送咨询消息（提醒咨询师）

### 7.5 基础后台管理与统计
**管理员端**
- 用户管理：学生/咨询师账号 CRUD、启禁用、重置密码
- 记录管理：咨询记录、预约记录、测评记录（列表检索/详情查看）
- 统计概览（不含学院维度）：咨询量趋势、预约量趋势、测评人数与风险等级分布

**验收标准**
- 管理员能对核心业务数据进行检索与监管操作
- 统计数据与业务数据一致（同源计算或一致的聚合口径）

## 8. 核心业务流程（文字版）
### 8.1 咨询流程
学生发起咨询（生成线程） → 咨询师查看线程并回复 → 学生追问（多轮） → 咨询师结束线程 → 管理员可监管与下架

### 8.2 测评流程
学生选择量表 → 答题提交 → 后端评分与分级 → 返回结果与建议 → 记录入库 → 学生/咨询师/管理员按权限查看

### 8.3 预约流程
咨询师配置排班 → 学生选择时段并提交预约 → 系统自动确认并占用 → 双方收到站内通知 → 咨询师完成预约 → 记录可追溯

## 9. 数据模型概览（实体级）
- 用户 User：账号、密码Hash、角色、状态、基础信息
- 学生档案 StudentProfile（可选）：姓名/学号等（与匿名展示解耦）
- 咨询师档案 CounselorProfile：姓名、职称、简介、擅长方向
- 咨询线程 ConsultThread：学生ID、咨询师ID、状态、创建/更新时间
- 咨询消息 ConsultMessage：线程ID、发送方、内容、时间
- 量表 Scale：PHQ-9 / GAD-7（固定）
- 题目 ScaleQuestion 与选项/分值（固定）
- 测评记录 Assessment：用户ID、量表ID、总分、等级、建议、时间
- 排班 ScheduleSlot：咨询师ID、日期、时间段、状态
- 预约 Appointment：学生ID、咨询师ID、时段ID、状态、时间
- 站内通知 Notification：接收者ID、类型、内容、已读状态、时间；预约类通知额外包含用户名称与咨询师名称
- 审计日志 AuditLog：操作者ID、动作、对象、时间、结果

## 10. 非功能需求
### 10.1 安全与隐私
- 最小权限原则：接口鉴权 + 角色授权 + 数据范围控制
- 敏感信息最小化展示：学生端不展示其他学生信息；咨询师仅在关联详情可见学生身份
- 审计留痕：关键增删改操作需记录

### 10.2 性能与可用性
- 本地演示环境下，核心列表查询与提交操作稳定可用
- 数据一致性优先：预约占用需保证不重复（事务/唯一约束）

## 11. 里程碑计划（15 周）
- 第 1–2 周：需求细化、原型/路由设计、数据库初稿
- 第 3–8 周：核心模块开发（登录权限、咨询、测评、预约、通知）+ 单元/接口自测
- 第 9–10 周：集成测试、权限与数据范围校验、性能与体验优化
- 第 11–13 周：论文初稿撰写与图表补齐（架构、流程、ER、测试）
- 第 14–15 周：论文定稿、答辩材料（PPT、演示视频、部署说明）

## 12. 风险与对策
- 数据敏感与合规：通过权限控制、审计留痕与最小化展示降低风险
- 时间不足导致范围膨胀：严格按“5.2 不做项”控制范围，优先保障三条闭环可演示
